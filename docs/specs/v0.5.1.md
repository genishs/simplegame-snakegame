# v0.5.1 Spec

## Goal
튜토리얼과 Stage 1의 틱 속도를 추가 완화하고, Stage 2/3 데이터를 신규 추가하여 Stage 1→2→3 자연 전이를 활성화한다. 보드 크기(20×20)와 시각·조작은 변경하지 않는다.

## User Stories
- 초보 플레이어로서, 튜토리얼이 더 여유로워서 조작을 충분히 익힐 시간을 갖고 싶다.
- 초보 플레이어로서, Stage 1도 충분히 느려서 처음 넘어가는 문턱이 낮았으면 한다.
- 플레이어로서, 사과를 일정 개수 먹으면 다음 스테이지로 자연스럽게 넘어가는 진행감을 느끼고 싶다.
- 숙련 플레이어로서, Stage 2·3에서 점점 빨라지는 도전감을 느끼고 싶다.

## Acceptance Criteria

### 틱 속도 변경
- [ ] 튜토리얼 틱이 350ms → **380ms**로 변경된다 (`STAGES[0].tick === 380`)
- [ ] Stage 1 틱이 110ms → **140ms**로 변경된다 (`STAGES[1].tick === 140`)

### Stage 1 클리어 조건 추가
- [ ] Stage 1에 `clearAfterApples: 5`가 부여된다 — 사과 5개를 먹으면 "Stage 1 클리어!" 안내 후 Stage 2로 전환된다
- [ ] Stage 1 클리어 안내 문구는 "Stage 1 클리어!" / "곧 Stage 2로 이동합니다" 이다

### Stage 2 신규 추가
- [ ] `STAGES` 배열 세 번째 항목으로 Stage 2가 추가된다
  - `id: 2`, `label: "Stage 2"`, `cols: 20`, `rows: 20`, `tick: 130`, `snakeLen: 3`, `clearAfterApples: 5`, `noFailOnHit: false`
- [ ] Stage 2에서 사과 5개를 먹으면 "Stage 2 클리어!" 안내 후 Stage 3으로 전환된다

### Stage 3 신규 추가
- [ ] `STAGES` 배열 네 번째 항목으로 Stage 3이 추가된다
  - `id: 3`, `label: "Stage 3"`, `cols: 20`, `rows: 20`, `tick: 120`, `snakeLen: 3`, `clearAfterApples: null`, `noFailOnHit: false`
- [ ] Stage 3은 `clearAfterApples: null` — 이후 끝없이 이어지는 엔드리스 모드로 동작한다

### 전이 흐름
- [ ] Tutorial → Stage 1 전이는 v0.5와 동일한 흐름(사과 3개 → STAGE_CLEAR 800ms 홀드 → Stage 1)을 유지한다
- [ ] Stage 1 → Stage 2 → Stage 3 전이는 기존 `advanceStage()` 로직을 그대로 사용한다 (코드 추가 없이 데이터만으로 동작해야 한다)
- [ ] Stage 3에서 사과를 계속 먹어도 `advanceStage()`가 실행되지 않고 Stage 3이 유지된다 (기존 `stageIndex >= STAGES.length` 분기 확인)

### HUD 및 시각
- [ ] 각 스테이지 전환 시 HUD "Stage:" 레이블이 해당 스테이지 `label` 값으로 업데이트된다
- [ ] 보드 크기(20×20), 팔레트, 뱀·사과 드로잉은 변경되지 않는다

### 버전 관리
- [ ] `VERSION` 파일이 `0.5.1`로 갱신된다
- [ ] 페이지(`index.html`) 표시 버전이 `v0.5.1`로 갱신된다
- [ ] `HISTORY.md`에 v0.5.1 항목이 추가된다

## Out of Scope
- 보드 크기(20×20) 변경
- 시각 토큰·팔레트·애니메이션 변경
- 조작 키 매핑 변경
- BLOCKED / noFailOnHit 동작 변경
- Stage 4 이상 스테이지 데이터 추가
- 스테이지 간 개별 클리어 안내 문구의 다국어화

## Non-functional notes
- 페이로드 예산 50KB 유지 — STAGES 항목 2개 추가는 수 바이트 수준이므로 예산 여유 있음
- 틱 속도 변경은 `STAGES` 배열 리터럴만 수정하면 되며, 게임 루프(`loop()` RAF 내 `tickAccum` 누산) 로직은 건드리지 않아도 자동 반영된다
- `clearAfterApples` 비교 분기는 기존 `tick()` 내 `stage.clearAfterApples != null && applesEaten >= stage.clearAfterApples` 조건을 그대로 재사용한다 — 새 분기 코드 불필요
- Stage 3 엔드리스 보장: `advanceStage()` 내 `stageIndex >= STAGES.length` 가드가 이미 있으므로 추가 처리 없음

## Decisions
- **(A) 자연 전이 즉시 활성화 채택** — Stage 1·2에 `clearAfterApples` 값을 부여한다. 배열에 데이터만 추가하고 전이를 막으면 Stage 2·3은 영구 dead code가 되어 검증조차 불가능하다. 기존 `advanceStage()` 설계가 "스테이지는 데이터, 전이는 로직"을 의도하고 있으므로 (HISTORY v0.4 참조) 데이터 추가만으로 전이를 활성화하는 것이 설계 의도에 부합한다. stages 1–3은 여전히 매우 쉬워야 한다는 프로젝트 규칙(memory: project_stage_difficulty)도 140ms/130ms/120ms 범위에서 충족된다.
